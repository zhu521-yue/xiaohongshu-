## 1. 你到底在建什么(全景)

### 1.1 一句话定义

一个用 **LangGraph 编排**、底层复用 **Spider_XHS 平台接口**、由**国产大模型**驱动的小红书内容运营闭环系统。它把"选主题 → 找痛点 → 生成内容 → 合规 → 人工确认 → 发布 → 复盘 → 沉淀记忆"做成一条可回退、可审核、可复盘的流水线。

### 1.2 两阶段的本质区别

| | 阶段一(冷启动) | 阶段二(成长+变现) |
|---|---|---|
| 只做 | 知识分享 | 知识分享 70% + 软广 30% |
| 软广 | **硬禁止** | 允许,但低频、不连发、强制人工审 |
| 进入条件 | 默认起点 | 累计≥30篇 + 3个稳定高表现主题 + 评论区出现购买意图 |
| 新增能力 | — | 千帆选品、软广生成、蒲公英达人 |

阶段切换由 `check_account_stage` 节点用规则判断,**不是按时间硬切**。

### 1.3 最终系统的形态(你要建成的样子)

```text
用户输入主题
   ↓
[规则] 判断账号阶段(cold_start / growth / monetization_ready)
   ↓
[采集] Spider_XHS 搜同主题笔记 + 抓评论   ← 真实数据来源
   ↓
[LLM] 提取用户痛点、拆解爆款标题结构
   ↓
[记忆] GraphRAG 检索历史高表现经验
   ↓
[LLM] 内容策略判断(今天发什么、图文还是视频)
   ↓
[LLM] 内容生成(图文 / 视频脚本 / 软广)
   ↓
[LLM+规则] 合规审核(低→人工 / 中→改 / 高→停)
   ↓
[人工] 强制确认节点
   ↓
[工具] Spider_XHS 创作者平台发布          ← 写入操作,高风险
   ↓
[采集] 回收发布表现数据
   ↓
[LLM] 复盘 → 回写 GraphRAG
   ↓
进入下一轮
```

### 1.4 三份材料如何拼成一个系统

- **业务方案** = 每个节点"输出什么内容、遵守什么规范"(看第 11~17 节)。
- **技术方案** = 这些节点"怎么用 LangGraph 串起来"(看 State / 路由 / 子图)。
- **Spider_XHS** = `采集` 和 `发布` 两类节点背后真正干活的接口。

> 你的工作 = 写 LangGraph 的节点函数,在"采集/发布"节点里调用 Spider_XHS,在"分析/生成/合规/复盘"节点里调用国产 LLM,用业务方案的规范约束输出。

## 2. 前置认知(动手前必须懂的 4 件事)

### 2.1 LangGraph 是什么角色

它是**流程编排器**,不是"让 AI 自由发挥"的东西。核心三件套:
- **State**:一个贯穿全流程的数据字典(技术方案 §7 已给出完整字段)。每个节点读 State、改 State。
- **Node(节点)**:一个普通函数,输入 State、输出对 State 的更新。
- **Edge / Conditional Edge(边 / 条件边)**:决定下一个去哪个节点。条件边 = 路由函数(技术方案 §8 已给出 4 个路由)。

> 记住:**规则节点管边界、LLM 节点管生成、工具节点管平台、人工节点管确认、GraphRAG 管记忆、LangGraph 管串联。**

### 2.2 Spider_XHS 是什么、能给你什么

它是别人逆向封装好的小红书接口库,**靠 Cookie 鉴权 + Node.js 跑签名 JS**。你直接复用它的函数,不用碰签名细节。它给你的能力(已逐个核对):

| 平台 | 类(文件) | 阶段一要用的 | 阶段二再用的 |
|---|---|---|---|
| PC 采集 | `XHS_Apis`(`apis/xhs_pc_apis.py`) | `search_some_note`、`get_note_all_comment`、`get_note_info`、`get_search_keyword` | `get_all_metions`、`get_all_likesAndcollects`(看互动) |
| 创作者 | `XHS_Creator_Apis`(`apis/xhs_creator_apis.py`) | `post_note`、`get_all_publish_note_info` | 同左 |
| 千帆(选品) | `QianFanAPI`(`apis/xhs_qianfan_apis.py`) | — | `get_some_user`、`get_user_detail`、`get_user_shop`、`get_user_item` |
| 蒲公英(达人) | `PuGongYingAPI`(`apis/xhs_pugongying_apis.py`) | — | `get_some_user`、`get_user_detail`、`get_user_fans_detail`、`send_invite` |

辅助:`xhs_utils/common_util.py` 的 `init()` 会读 `.env` 里的 `COOKIES` 并返回 `(cookies_str, base_path)`;`xhs_utils/data_util.py` 有 `handle_note_info`(清洗笔记)、`save_to_xlsx`(存表)。

### 2.3 国产 LLM 怎么接

三家都兼容 OpenAI 风格调用,任选其一起步(后面可换):
- **通义千问(DashScope)**:`OpenAI` 兼容端点,模型名如 `qwen-plus` / `qwen-max`。
- **智谱 GLM**:模型名如 `glm-4`。
- **DeepSeek**:`OpenAI` 兼容端点,模型名 `deepseek-chat`。

> 建议:**把 LLM 调用封装成一个 `llm_client`,只在一处配置厂商和 Key**。这样后面换模型只改一个文件。LangGraph 生态可用 `langchain-openai`(指向兼容端点)或各家官方 SDK。

### 2.4 ⚠️ 你选了"采集+发布都接",这条路的红线(必须先认同再开工)

你的两份方案都建议"第一版不接自动爬虫和自动发布",Spider_XHS 的 README 也写明 Cookie 有时效、有封号风险。你选择都接,我尊重,但**这条路必须钉死以下安全基线**,否则大概率封号:

1. **先只读、后写入**:M2 只做采集(读),M4 才做发布(写)。不要跳着来。
2. **发布前强制人工确认**:`human_review` 是硬节点,任何内容不经人工点头不发。
3. **低频 + 拟人**:采集/发布之间加随机延时(几秒~几十秒),单日发布量个位数,不要批量。
4. **Cookie 单独存、不进 Git**:放 `.env`,加进 `.gitignore`。
5. **采集只存主题级数据**:评论去标识化,不存昵称/头像/主页/用户 ID(业务方案 §19.3)。
6. **内容合规**:母婴/健康/教育/金融四类,禁绝对化、禁疗效承诺、禁夸大(§19.2)。
7. **失败即停**:接口返回 `success=False` 或风控提示,立即停手排查,不要重试轰炸。

> 这 7 条是开发约束,写进每个采集/发布节点。第 8 章有完整 checklist。

## 5. 接入 Spider_XHS 作为能力底座

### 5.1 引入方式(二选一)

- **方式 A(推荐起步):整目录复用**。让你的项目能 `import` 到 `Spider_XHS-master` 下的 `apis/` 和 `xhs_utils/`。最省事,签名 JS、`static/`、`.env` 都现成。
- **方式 B:抽成内部包**。把 `apis/`、`xhs_utils/`、`static/` 拷进你项目,改成自己的模块。干净但要自己维护签名 JS 更新。

> 起步用 A。等系统稳定再考虑 B。

### 5.2 用"薄封装"隔离(关键设计)

**不要在 LangGraph 节点里直接调 Spider_XHS**。在 `platforms/` 里包一层,节点只调你的封装。好处:Spider_XHS 升级、换库、加延时/重试/去标识化,只改 `platforms/`,节点不动。

每个封装该做的事:
- `platforms/collector.py`:包 `XHS_Apis`。对外提供"按主题搜 N 篇笔记"、"抓某篇笔记评论"。**内部统一加随机延时、做评论去标识化、失败返回结构化错误**。
- `platforms/creator.py`:包 `XHS_Creator_Apis`。对外提供"发布图文/视频"、"拉已发布列表"。**内部强制检查"是否已人工确认"标记**。
- `platforms/qianfan.py` / `pugongying.py`:阶段二再写。

### 5.3 采集/发布对应的函数清单(照着调,别找错)

**采集(阶段一就用)**
- 按主题找笔记:`XHS_Apis.search_some_note(query, require_num, cookies_str, sort_type_choice=...)`
  - `sort_type_choice`:0综合 / 1最新 / 2最多点赞 / 3最多评论 / 4最多收藏。做痛点分析建议用 2 或 4(高表现样本)。
- 笔记详情:`XHS_Apis.get_note_info(note_url, cookies_str)`
- 全部评论:`XHS_Apis.get_note_all_comment(note_url, cookies_str)` ← 痛点的金矿
- 清洗笔记:`xhs_utils.data_util.handle_note_info(note_info)`

**发布(M4 才用)**
- 发图文/视频:`XHS_Creator_Apis.post_note(noteInfo, cookies_str)`
  - `noteInfo` 关键字段:`title`、`desc`、`media_type`("image"/"video")、`images`(图片字节列表,最多15张)或 `video`(视频字节)、`topics`(话题列表)、`type`(0公开/1私密)、`postTime`(定时,None=立即)。
  - **起步技巧**:先用 `type=1`(私密)发一篇测试,确认链路通且内容正确,再改公开。
- 已发布列表:`XHS_Creator_Apis.get_all_publish_note_info(cookies_str)` ← 复盘取数

### 5.4 安全顺序(再强调)

```text
M2 只读采集(search + comment)
   ↓ 跑稳了
M4 写入发布(先私密 type=1 测 → 再公开)
```

## 6. 开发里程碑路线图(主线)

> 每个里程碑:**目标 → 做什么 → 完成标志**。达标才进下一个。

### M0 · 环境与链路验证
- **目标**:证明三条链路(Node 签名 / LLM / Cookie)都通。
- **做什么**:完成第 3 章全部清单。
- **完成标志**:能搜到真实笔记 + LLM 能回话 + Cookie 自检通过。

### M1 · 内容生成最小闭环(不接平台,纯本地)
- **目标**:跑通"主题 → 痛点(先用假数据/手填)→ 策略 → 生成 → 合规 → 人工确认 → 输出 Markdown"。
- **做什么**:
  - [ ] 写 `state.py`、`config.py`、`llm/client.py`。
  - [ ] 写节点:`input` → `stage`(默认 cold_start)→ `strategy` → `content`/`video` → `compliance` → `human_review` → 存 `output/`。
  - [ ] 痛点先用**手工填的假数据**,不接采集。
  - [ ] 在 `graph.py` 用 §8 的 4 个路由把图连起来,`main.py` 跑一轮。
- **完成标志**:输入一个主题,`output/` 里生成一篇带标题/正文/图片文案/标签/合规提醒的图文 Markdown,且 cold_start 阶段**绝不**走软广分支。

### M2 · 接入只读采集(痛点用真实数据)
- **目标**:把 M1 里的假痛点换成 Spider_XHS 采集的真实数据。
- **做什么**:
  - [ ] 写 `platforms/collector.py`(含随机延时 + 评论去标识化 + 错误兜底)。
  - [ ] 改造 `insight_node`:调 `collector` 搜同主题高表现笔记 + 抓评论 → 喂给 LLM 提痛点(用 §6.3 的痛点 Prompt)。
  - [ ] 采集结果落 `data/`(笔记表/评论表,结构抄业务方案 §15.1/§15.2)。
- **完成标志**:输入主题后,系统能展示"从真实评论里提炼出的 Top 痛点 + 排序",且数据已去标识化、采集有延时。

### M3 · 复盘闭环 + 运营记忆雏形
- **目标**:让系统"记住"哪些主题/标题/形式效果好。
- **做什么**:
  - [ ] 写 `review_node`:输入发布表现(此阶段**手工录入**曝光/点赞/收藏/评论)→ LLM 复盘(用 §14.6 复盘 Prompt)。

## 8. 安全与合规红线(开发约束 checklist)

> 这是你选"采集+发布都接"必须落地的护栏。每条都要在代码里有对应实现。

**平台操作**
- [ ] 采集/发布之间有随机延时(几秒~几十秒)。
- [ ] 单日发布量个位数,绝不批量。
- [ ] 接口返回 `success=False`/风控提示 → 立即停,不重试轰炸。
- [ ] Cookie 在 `.env`,已进 `.gitignore`;失效有重取流程。
- [ ] 发布默认先 `type=1` 私密验证,再切公开。

**人工关卡**
- [ ] `human_review` 是硬节点,无确认不发布。
- [ ] 软广、母婴/健康/教育/金融内容,人工审更严。

**内容合规**(母婴/健康/教育/金融尤甚)
- [ ] 禁绝对化表达、禁疗效承诺、禁夸大效果、禁制造焦虑、禁虚构经历。
- [ ] 商业推广加广告/合作标识。
- [ ] 合规节点判 high → 直接停发。

**数据隐私**
- [ ] 评论去标识化:不存昵称/头像/主页/用户 ID。
- [ ] 只做主题级/内容级分析,不做个人画像追踪。

**阶段二专属硬规则**
- [ ] cold_start 阶段 `allow_soft_ad=False`,代码层拦死。
- [ ] 不连发软广;单周软广 ≤ 2 篇。

## 9. 调试与验证方法

- **单节点先单测**:每写完一个节点,用假 State 直接调它,确认进出字段对,再连进图。
- **分阶段编译图**:M1 只连到 `human_review`+存 Markdown;别一上来连发布。
- **LangGraph 可视化**:编译后导出图结构图,肉眼检查边连对没有(尤其条件边的分支映射)。
- **干跑发布**:M4 初期让 `publish_node` 只打印 `noteInfo` 不真发,确认参数齐全,再放开真发。
- **采集小批量**:`require_num` 先设 3~5,验证解析正确再加量。
- **Cookie 自检前置**:每次跑全流程前先跑一次 Cookie 有效性检查,避免半路失效白跑。

## 10. 常见坑与排查

| 现象 | 大概率原因 | 处理 |
|---|---|---|
| 采集/发布报签名相关错误 | Node.js 没装好 / `static/` JS 路径不对 | 确认 `node -v` ≥20,确认 PyExecJS 能跑通 `static/` 下脚本 |
| 接口返回 `success=False` 且像风控 | Cookie 失效 / 操作太频繁 | 重取 Cookie;加大延时;停手观察 |
| `post_note` 图片报错 | `images` 要的是**字节**不是路径 | 读成 bytes 再传(参考 `xhs_creator_apis.py` 示例) |
| LLM 输出不是合法 JSON | Prompt 没强约束 / 模型发挥 | Prompt 里强制"只输出 JSON";用 pydantic 校验+重试 |
| cold_start 却生成了软广 | 路由/规则漏判 | 检查 `route_content_type` 与 `allow_soft_ad` 拦截 |
| 视频发布卡住 | 转码轮询未就绪 | `post_note` 内有转码轮询,耐心等;失败看 `query_transcode` 返回 |
